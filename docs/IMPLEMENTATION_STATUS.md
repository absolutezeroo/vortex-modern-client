## Recent Work Recorded

- 🟡 **Further bugs found live in the same debugging session (input bubbling, skin rendering, PixiJS renderer setup)**:
  - **`HeliumApp`'s mouse-wheel handler never bubbled**: `WindowController.update()` returns `false` for `WHEEL` by default — only `ItemListController`/`ScrollBarController` override it to actually scroll. Our handler dispatched the `WHEEL` event straight at `findWindowAtPoint()`'s result (the deepest/leaf window under the cursor — inside a scrollable box, usually a list item/icon/label, not the scrollable container itself) and stopped, so scrolling only worked on the rare occasion the cursor happened to be directly over the scrollable window itself. Fixed by walking up the `parent` chain (mirroring `MouseEventProcessor.passMouseEvent()`'s bubbling for other mouse events) re-dispatching at each ancestor until one actually reports `handled`.
  - **PixiJS `multiView` not enabled**: several widgets read pixels back off-screen via `renderer.extract.canvas()` (`AvatarImageWidget` avatar/badge previews, `RoomEngine.pixiTextureToCanvas()` room icons, `RoomRenderingCanvas.takeScreenShot()`) — each of those is, from WebGL's point of view, "another canvas" sharing the app's GL context. Without `multiView`, PixiJS's `GlContextSystem.ensureCanvasSize()` warns on every one of those extractions ("multiView is disabled, but targetCanvas is not the main canvas") because it only expects the single view canvas. Set `multiView: true` on the `Application.init()` config in `Helium.ts` — PixiJS's documented flag for exactly this one-context/several-canvases case.
  - **`BitmapSkinParser`/`HabboWindowManager` renderer cache key too coarse, and descriptor `states` ignored**: AS3's `parseSkinDescription()` prefers the caller-supplied `states` (the element descriptor's own `<states>`) and only falls back to the skin XML's own `<states>` when that list is empty — our `BitmapSkinParser.parse()` never accepted a states override at all, which silently broke skins like `habbo_skin_icon_set` whose own top-level `states` is empty (every icon style's state→layout/template mapping instead lives entirely on its element descriptor). Compounding this, `HabboWindowManager`'s renderer cache was keyed by `layout` filter alone, but several descriptors share the same `layout` (e.g. `icon16` reused across arrow styles 0-9) while pointing at different `states`/`template` — so the first-parsed descriptor's renderer silently got reused for every other descriptor sharing that layout name. Fixed by adding an optional `descriptorStates` parameter to `parse()` (AS3-priority: descriptor states first, XML states as fallback) and folding the descriptor's own states into the cache key (`` `${layoutFilter}::${JSON.stringify(descriptor.states ?? [])}` ``).
  - **`BitmapSkinRenderer.draw()` stretched pieces that should never scale**: AS3 only resizes a drawn piece when its scale mode actually calls for it (`STRETCH`/`TILED` on either axis); `FIXED`/`MOVE`/`CENTER` always blit the piece at its own native size via `BitmapData.copyPixels()`, never stretched to fill the layout region's authored box, even when that box is a different size than the source crop (e.g. a small icon glyph centered in a bigger padded box). Our port instead sized the draw off `destW === srcW`/`destH === srcH`, which stretched every `FIXED`-mode entity whose region size didn't happen to match its source crop. Fixed by branching on the actual scale mode (`tiled` / `stretched` / neither) instead of comparing dimensions, matching AS3's `_loc16_`/`_loc9_` logic in `BitmapSkinRenderer.as::draw()`.
- 🟡 **Catalog icon/preview rendering bugs found and fixed live (dev-environment debugging session)**: three separate, previously-invisible bugs, each masking the next.
  - **`BitmapFileLoader.dispose()` premature `Assets.unload()`**: `AssetLibrary.handleAssetLoadEvent()` disposes the loader as routine cleanup right after handing its loaded `Texture` off to a new `BitmapDataAsset` (which caches it under an asset name for reuse). `dispose()` was calling `Assets.unload(url)`, telling PixiJS's `Assets` cache to release the GPU resource almost immediately — so a furniture/product icon rendered once (before/during the async unload) but broke on every later reuse of that cached asset (icon disappears on a second click; a catalog page loading many icons at once hit the race on nearly all of them). Fixed by dropping the local `_texture` reference only; the `BitmapDataAsset` now owns the texture's lifecycle from there (its own `dispose()` already calls `texture.destroy(true)`).
  - **`RoomEngine`/`AvatarImageWidget` texture→canvas conversion fragility**: both read `texture.source.resource` directly and `drawImage()`'d it — but PixiJS doesn't guarantee a CPU-side resource stays attached to a `TextureSource` once uploaded to the GPU (`resource` can legitimately be `undefined` for a fully valid on-screen texture). Switched both to `renderer.extract.canvas(texture)`, PixiJS's own supported readback API.
  - **`ProductGridItem.setIconImage()` use-after-close**: AS3 (`ProductGridItem.as::setIconImage()`) copies the source `BitmapData`'s pixels into a buffer the target *already owns* (`copyPixels()`), so disposing the source parameter afterward never touches what's on screen. Our port instead did `target.bitmap = image` (a reference assignment, not a copy) and then unconditionally called `image.close()` on that *same* object when `disposeSource` was true — destroying the bitmap the target had just been handed. This was the actual root cause behind "icon shows once then disappears" surviving even after the `BitmapFileLoader` fix above. Fixed: only close the source when it was *not* handed to a live target.
  - **`ProductViewCatalogWidget`'s `product_image_widget` never hidden**: AS3 finds this child (`var_1030`), unconditionally hides it in `init()` and again before every preview render, only re-showing it for the one product type its own preview switch doesn't otherwise handle (`ProductDisplayWrapper.isSupported()` — in practice only `chat_style`, since `"r"` has its own earlier case in the same switch and is therefore unreachable there despite `isSupported()` accepting it too). Our port never looked this node up at all, so its `placeholder_image` (visible by default per `ProductImageWidget`'s constructor) stayed on screen permanently, overlapping the room-canvas furniture/wall-item preview. Fixed by porting the lookup/hide-on-init/hide-before-render steps, and also implemented the previously-stubbed `ProductDisplayWrapper` (`habbo/catalog/viewer/ProductDisplayWrapper.ts` — a small `IProductDisplayInfo` adapter over `IProduct`, per the primary `win63_2026_crypted_version` source which adds `"r"` support and `botFigureString`/`extraData` over the secondary source's chat-style-only version) to wire up that default case for real.
- 🟡 **`LocalizationCatalogWidget.retrieveCatalogImage()` fully ported** (network fallback for a catalog page's localized images not already in the local asset library - was just logging and skipping): now calls the real `IAssetLibrary.loadAssetFromFile()` (already used elsewhere, e.g. `AvatarAssetDownloadLibrary`), builds the URL from `image.library.catalogue.url` or `image.library.url + "Top_Story_Images/"` depending on whether the target element carries a `TOP_STORY` tag (matching AS3 exactly), and applies the image via the existing `setElementImage()` once loaded. One AS3-fidelity note: AS3 listens via `addEventListener("AssetLoaderEventComplete", ...)`, but our `AssetLoaderStruct.dispatchEvent()` always emits under the generic `'event'` key regardless of the event's own `.type` - so a literal translation would silently never fire. Used the already-established, actually-working pattern instead (`loader.events.on('event', event => { if(event.type === AssetLoaderEventType.COMPLETE) ... })`, same as `AvatarAssetDownloadLibrary`/`EffectAssetDownloadLibrary`).
- 🟡 **`UniqueLimitedItemWidget` ported and registered** (the "X left of Y" LTD supply overlay on the product view page): was entirely missing - not implemented, not registered in `CatalogPage.createWidget()`'s switch (`'limitedItemWidget'` case). Its layout container (`limitedItemWidget` -> `unique_item_overlay_container`, `widget_type="limited_item_overlay_supply"`) has no default-hidden attribute in the compiled layout - AS3's widget class is the only thing that ever calls `window.visible = false`, so with the widget never constructed it showed constantly regardless of the selected product, which is what surfaced this gap. Full port: shows only for `pricing_model_single` + `isUniqueLimitedItem` offers, sets `supplyLeft`/`seriesSize` on the existing `LimitedItemSupplyLeftOverlayWidget`, and refreshes every 20s via a new real `HabboCatalog.sendGetProductOffer()` (new `GetProductOfferComposer`, message ID `1692` - cross-checked against both `win63_version` (id `3691`, a different build's numbering) and the primary `win63_2026_crypted_version` manifest (`1692`), which wins per the project's source-priority rule).

  - Known gap, tagged `TODO(AS3)` in `HabboClubCenter.ts::initComponent()`: the "earn credits via video" offer button (`btn_earn`) has no real backing system — `IOfferCenter`/`IOfferExtension` don't exist anywhere (`catalog/offers/` is empty) and `IHabboCatalog.getOfferCenter()`/`getHabboClubOffers()` are stubs. AS3 itself gates this button behind `offers.enabled`+`offers.habboclub.enabled` config flags, so it stays hidden exactly as it would with those flags off — not an invented workaround, just inert until the offer-center system is ported.
- 🟡 **`habbo/freeflowchat` viewer/style layer ported** — this is a separate, standalone chat-bubble system from the wired-up `habbo/ui/widget/roomchat` one (different catalog: `chatstyles_xml` vs `roomchat_styles_chatstyles_xml`; freeflowchat itself still isn't wired into `RoomUI`/`RoomDesktop`, see the room-tools row above).
  - `ChatStyleLibrary`/`ChatStyle`/`IChatStyle`/`IChatStyleLibrary`/`IChatStyleInternal`/`BlankStyle` ported in full, including the `regpoints` config-text mini-parser (`hasConfig`/`getConfigCSV`/`getConfigPoint`/`getConfigRect`/`getConfigBoolean`) and the background-tint + nine-slice pipeline. `HabboFreeFlowChat.ts` gained the two AS3 static helpers (`createNineSliceSprite`/`createPixelPerfectNineSliceSprite`, both backed by PixiJS's native `NineSliceSprite` — AS3's "manual" BitmapData-baking variant, `ManualNineSliceSprite`, was still ported as its own class for fidelity, but its GPU output is identical) plus real `roomEngine`/`localizations` DI wiring (previously absent) and a `chatStyleLibrary`/`roomChatBorderLimited` passthrough.
  - `ChatBubbleFactory` (the freeflowchat one, distinct from `ui/widget/roomchat/style/ChatBubbleFactory`) ported: avatar/pet image cache + LRU-style eviction (`imageReady`/`imageFailed`/`avatarImageReady`/`petImageReady`/`discardOldBitmaps` — including AS3's quirk of never clearing the disposal array outside `dispose()`), the full `applySpecialChatContent()` localization-driven special messages (respect/pet-respect/pet-treat/ping/handitem/mutetime/pet-revive-refertilize-speedfertilize), `isSystemNotificationChat()`, `resolveRoomUserName()`, and bubble pooling (`getNewChatBubble`/`getNewEmptySpace`/`recycle`).
  - `TODO(AS3)` scope cuts, all commented in-code: `getUserImage()`/`getPetImage()` always return null (same gap as `@habbo/ui/handler/ChatWidgetHandler.ts` — no Texture→ImageBitmap conversion utility, no ported `HabboFaceFocuser`, no `IRoomEngine.getPetImage()`/`PetFigureData`). `PooledChatBubble` (618 AS3 lines), `ChatHistoryEntryBitmapBubble`, and `ChatHistoryRoomChangeEntry` are structural stubs — real fields/constructor shapes so the factory has something real to build and pool, but their actual visual composition (`ChatBubble.as`, 511 lines, and the two history-entry classes' `TextField`-to-bitmap rendering) isn't ported; that's the bulk of the "chat display widget, not started" scope noted in the room-tools row above. `getHistoryRoomChangeEntry()`'s AS3 parameter type is an obfuscated identifier unrecoverable from any of the three source trees (its own file lives under the excluded `src/unknowns/` bundle) — modeled locally as a minimal `RoomChangeData { roomName }` shape, the only field actually consumed.
