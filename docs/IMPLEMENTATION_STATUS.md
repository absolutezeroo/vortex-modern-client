## Recent Work Recorded

- ✅ **Furni inventory preview panel: z-order and missing name/description fixed**:
  - `RoomPreviewerWidget`'s mini room canvas and the main room view's canvas both get parented directly onto the same shared PixiJS stage (see `RoomEngine.createRoomCanvas()`) — their relative stacking depends purely on PixiJS child-add order, not window z-order. Entering a room while the inventory preview was already open recreated the main room canvas *after* the preview's, pushing it on top and visually burying the preview wherever their screen rects overlapped. Fixed by re-asserting front-of-stage every frame in `syncCanvasPosition()` (which already runs continuously via the shared ticker).
  - `GroupItem.name`/`description` were computed once in `push()` and cached — including when empty. `FurnitureDataParser` injects the `roomItem.name.<id>`/`wallItem.name.<id>` localization keys these read from, but nothing guarantees that finishes before the first inventory items arrive and call `push()` (a race that doesn't exist in AS3, where furnidata + external texts load strictly before inventory contents). Once cached empty, name/description stayed blank forever. Now the getters retry the lookup lazily on read instead of trusting the one-time push() computation.

Also wired the previously-inert `flags:uint` constructor parameter (`HabboConfigurationFlags.SKIP_EXTERNAL_VARIABLES`/`SKIP_LOCALIZATIONS`) through `Component` → `CoreLocalizationManager` → `HabboLocalizationManager`/`HabboConfigurationManager`, including `HabboConfigurationManager.resetAll()`'s previously-missing final branch (immediate unlock+complete vs. eager `initConfigurationDownload()`) — behaviorally inert today since nothing passes non-default flags yet, but now matches the full AS3 member signature. `HabboConfigurationManager.as::parseArguments()` (flashvars/query-string config overrides from `(context as ICore).arguments`, `_`→`.` key normalization, then `clearArguments()`) was unported entirely — now implemented and wired into `resetAll()` at the correct call site; behaviorally inert today since nothing currently populates `ICore.arguments`, but no longer a silent gap. `communication/encryption` (ArcFour, DiffieHellman) and `communication/login` audited clean — no divergence (note: win63's `ArcFour.as` KSA swap is itself decompiler-corrupted, `sbox[j]=0` instead of completing the swap; the TS port already has the correct swap, don't "fix" it to match win63). `communication/messages/parser` loop-bearing files (79 of 339) partially audited, most of `communication/messages/incoming`/`outgoing` (600+ simple field composers) and 58 of the 79 loop-bearing parsers not yet covered — candidates for a follow-up pass.
- 🐛 **Third audit batch: 10 more AS3-divergence bugs found and fixed in `habbo/communication/messages/{incoming,outgoing}/{room,inventory}`** (targeted audit of the actual wire-format composers/parsers, not just the higher-level model/UI code):
  - `RoomDimmerChangeStateComposer.ts`/`RoomDimmerGetPresetsComposer.ts` sent an empty array — AS3 both take a single `objectId:int` (the dimmer furni's item id, confirmed via `FurnitureDimmerWidgetHandler.as` call sites) and send `[objectId]`. `RoomDimmerSavePresetComposer.ts` was missing 2 of AS3's 7 wire fields (a hardcoded `false` and the `objectId`). All three now take the id and match AS3's array shape; wired through `RoomSession.ts`'s previously-unused `_itemId` params. Zero live impact today (the dimmer UI widget itself isn't ported yet), but was blocking that port from ever working.
  - `MountPetComposer.ts`/`DismountPetComposer.ts` — AS3 has exactly ONE composer, `MountPetMessageComposer(petId, mount:Boolean)`; the TS port invented a split into two composers that both dropped the boolean entirely, relying on an assumption ("server toggles state") not in AS3. Merged back into one `MountPetComposer(petId, mount)`, deleted the invented `DismountPetComposer`.
  - `BanUserWithDurationMessageComposer.ts` — wrong field order (`[userId, banType, roomId]` vs AS3's `[userId, roomId, banType]`) **and** wrong type (`banType` retyped as a number with an invented 1/2/0 mapping; AS3 sends the literal ban-duration string). `RoomSession.banUserWithDuration()`'s hand-rolled string→number switch was removed; the AS3 string constant now flows straight through.
  - `TradingItemListMessageParser.ts` reused `FurniListItemParser` (the regular inventory item format) to parse trade-offer items — AS3 uses a completely different class (`class_3066`: adds `isGroupable` before `stuffData`, replaces the tradeable/sellable/rent-period flags with `creationDay`/`creationMonth`/`creationYear`). Created `TradingFurniItemParser.ts` matching `class_3066` field-for-field; every traded item was being misparsed from the first field onward.
  - `PetInventoryMessageParser.ts` was densely wrong: read `breedId`'s wire slot as a loop count, undercounted the custom-parts loop by 3x (AS3 reads 3 ints per custom part, TS read 1), invented two fields that don't exist on the wire (`hasCustomLayerPart`, `petRarity`), and hardcoded `level: 1` with a comment claiming "level comes in a separate message" — false, AS3 reads it as the very next field. Rewritten to match `class_2838`/`class_2486` exactly; the parser cursor was desynced from the first pet entry onward.
  - `BadgesMessageParser.ts` (inventory) never read `totalFragments`/`fragmentNo`, and split parsing into two invented phases (a badge-id-only loop, then a separate slotId+badgeId loop) instead of AS3's single `count` + `(slotId, badgeId)` loop. Rewritten to match `BadgesEventParser.as`; cross-checked against the already-correct `HabboUserBadgesMessageParser.ts` (users module), which has the right shape and confirmed this was a regression specific to the inventory-badges file.
  - `FurniListAddOrUpdateMessageParser.ts` — **live bug**: parsed exactly one item with a comment claiming "AS3 reads 1 item, but message may contain more" (backwards — AS3 reads a `count` then loops `count` times). `HabboInventory.onFurniListAddOrUpdate()` already correctly loops over `parser.items`, so this was silently dropping/corrupting every item after the first whenever the server batched more than one add/update in a single message.
  - `FurniListRemoveMultipleEventParser.as` (bulk furni removal, message ID 149) had no TS port at all — zero references anywhere in the engine. Fully ported: `FurniListRemoveMultipleMessageParser.ts`, `FurniListRemoveMultipleMessageEvent.ts`, registered in `HabboMessages.ts`, and wired into `HabboInventory.ts::onFurniListRemoveMultiple()` calling the new `FurniModel.removeFurnis()` (added to `FurniModel`/`IFurniModel`, matching AS3's `removeFurnis()`).
  - Audited clean, no divergence: `communication/messages/{incoming,outgoing}/navigator` + `newnavigator` (96 files — one dead-code constructor overload noted, not a bug), all `incoming/room` wrapper files (47), `outgoing/room/action`/`avatar`/`pet`(except Mount/Dismount)/`settings`/most of `furniture`(except the 3 RoomDimmer bugs), `outgoing/room/engine` (MoveAvatar/MoveObject/PlaceObject/PickupObject/GetFurnitureAliases).
  - Not yet covered (follow-up candidates): `outgoing/room/chat`/`session` (8 files), most of `incoming`/`outgoing` `inventory`/`users` beyond the item-list parsers above (~90 files, `BlockListMessageParser`/`IgnoredUsersMessageParser` flagged as the likely next risk), and `help`/`notifications`/`friendlist`/`moderation`/`quest`/`handshake`/`catalog`/etc. incoming/outgoing directories (untouched this pass).
