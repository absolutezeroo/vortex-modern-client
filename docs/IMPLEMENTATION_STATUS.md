## Recent Work Recorded

- ­¤ö┤ **Root-cause fix: `HabboWindowManager.buildFromXML()` passed the desktop as `parent` instead of `null`, unlike real AS3 - fixed at the source in `WindowParser`/`WindowContext`, not patched per call site.** User-reported symptom: a small unstyled grid-item-shaped badge (price text + currency icon) appeared fixed in the screen's top-left corner whenever a catalog page with an item grid loaded, and never went away, not even after closing the catalog; later found to reproduce for the chat style selector too. Found via `WindowDebuggerOverlay`'s picker + a new ancestor-chain readout (see below), which showed the orphan's parent was a bare `desktop_N`, not the real owning window.
  - **Investigation history (worth keeping - the first two fix attempts are real dead ends, not just noise)**: attempt 1 patched the 3 catalog call sites that build a `.clone()`-source template and never re-add it (`removeChild()` after construction) - fixed the reported symptom but left the same bug latent everywhere else `buildFromXML()` is called this way. Attempt 2 fixed `WindowParser`/`HabboWindowManager` to pass `parent = null` like AS3 (`HabboWindowManagerComponent.as::buildFromXML()`: `parseAndConstruct(param1, null, param3)`) - this is correct AS3 architecture (AS3's `WindowParser` constructor takes and stores its owning context independent of `parent`, which is why it can safely pass `null`; this port's `WindowParser` had no such field, so `buildFromXML()` was passing `desktop` as a workaround, with the side effect of auto-attaching every top-level built window to the desktop) - but shipping it alone broke room click-to-move, furniture clicks, and own-avatar clicks entirely, because `WindowContext.create()`'s `USE_PARENT_GRAPHIC_CONTEXT` (`0x10`) param-flag handling means a `null`-parent window with that flag routes to `_substituteParent` (a legitimate AS3 "holding pen" controller, `SubstituteParentController.ts`) instead of falling through to the desktop - and one specific caller (`RoomDesktopLayoutManager.setLayout()`, next bullet) was missing the explicit re-attach AS3 requires for exactly that case, which the *previous* desktop-fallback bug had been silently masking.
  - **The actual fix, in order**: (1) `WindowParser` constructor now takes and stores the owning `IWindowContext` (`core/window/utils/WindowParser.ts`), matching AS3's `WindowParser(context)` constructor - `parseSingleWindowEntity()` falls back to this stored context (`this._context`) instead of `parent.context` when there's no parent to read it off of. (2) `HabboWindowManager.buildFromXML()` now passes `null` as parent (was `desktop`), matching AS3 exactly. (3) `RoomDesktopLayoutManager.setLayout()` (`habbo/ui/RoomDesktopLayoutManager.ts`) - the ROOT of the *entire* room desktop UI (room view, infostand, room tools, chat input/chat widgets all nest inside it via `addRoomView()`/`addWidgetWindow()`) - was missing the explicit `desktop.addChild(this._layoutContainer)` call that real AS3 has right after building it (`DesktopLayoutManager.as::setLayout()` line 55: `(_layoutContainer.desktop as class_3151).addChild(_layoutContainer);` - confirmed via the clean `win63_2023_version` reference, since `win63_version`'s copy of this class is corrupted here). `room_desktop_layout_xml`'s root does carry the `0x10` flag (confirmed by inspecting the compiled JSON), so this one call site was a real, pre-existing gap that the old bug happened to paper over, not something introduced by fixing `WindowParser`.
  - **Audit method for the rest of the codebase**: rather than auditing all ~65 `buildFromXML()`/`buildWidgetLayout()` call sites by hand, wrote a one-off script checking every referenced compiled `window-layouts/*.json`'s root `params` for the `0x10` bit - only a root carrying that flag is even affected by the `parent=null` change (everything else still unconditionally falls through to `parent ?? this._desktop` inside `WindowContext.create()`, completely unchanged by this fix). Of the ~35 flagged layouts found this way, all but `room_desktop_layout_xml` already had a correct, explicit re-attach downstream (`rootWindow =` on an `IWidgetWindow` placeholder - used by all 26 files in `habbo/window/widgets/`, which internally does an unconditional `addChild()`; or `addGridItem()`/`addListItem()`/a direct `addChild()` by the constructing code's caller) - confirmed the shared `WindowController.addChild()`/list/grid-item-add implementations all correctly detach a child from whatever parent it already has (including `SubstituteParentController`) before reattaching, so these patterns are safe regardless of which parent a window started under.
  - Reverted the 3 catalog-specific `removeChild()` band-aids from attempt 1 once the real fix landed - they're redundant now (those templates simply never get attached anywhere to begin with) and AS3 has no equivalent of them.
  - **New debugger capability, not AS3-related** (`helium-client/src/debugger/WindowDebuggerOverlay.ts`): the tree inspector previously only showed a picked window's *descendants*; there was no way to see what it was actually parented under.
